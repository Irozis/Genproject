# Методология экспериментальной проверки подготовки рекламных макетов

## Цель эксперимента

Цель эксперимента — оценить, насколько приложение для автоматизированной подготовки рекламных графических материалов сокращает время работы, количество действий пользователя и число ручных правок по сравнению с ручным или шаблонным способом подготовки макетов.

Экспериментальный режим не является публичной функцией продукта. Он используется только для сбора обезличенных технических метрик в рамках ВКР.

## Сравниваемые методы

1. Ручной способ: подготовка макетов в графическом редакторе без автоматической адаптации.
2. Шаблонный способ: подготовка на основе заранее созданных шаблонов.
3. Приложение: подготовка через исследуемую систему автоматической генерации.
4. Другой способ: дополнительный метод, если он требуется в эксперименте.

## Исходные данные

Для каждого сценария фиксируются:

- один бренд;
- одно исходное изображение;
- набор рекламных форматов;
- при необходимости выбранная палитра;
- идентификатор участника без персональных данных.

Для изображения записываются только технические параметры: ширина, высота и соотношение сторон. Локальные пути к файлам, имена файлов и персональные данные не сохраняются.

## Сценарии

Small set:
- 5 форматов;
- 1 бренд;
- 1 изображение.

Medium set:
- 10 форматов;
- 1 бренд;
- 1 изображение;
- 1 палитра.

Large set:
- 20 форматов;
- 1 бренд;
- 1 изображение;
- 2-3 площадки.

Stress set:
- 320x50;
- 320x100;
- 728x90;
- 1456x180;
- 2880x300;
- 300x250;
- 1080x1080;
- 1080x1920.

Если точный формат отсутствует в каталоге, выбирается ближайший по размеру или соотношению сторон. Это отклонение фиксируется в отчете.

## Метрики

Основные метрики:

- общее время подготовки;
- среднее время на один формат;
- количество действий пользователя;
- среднее количество действий на формат;
- число выбранных форматов;
- число сгенерированных макетов;
- число экспортированных файлов;
- число предупреждений валидатора;
- число ошибок валидатора;
- число ручных правок;
- доля экспортированных форматов.

Сравнительные метрики:

- сокращение времени;
- сокращение количества действий;
- разница среднего времени на формат;
- разница количества ошибок;
- разница количества ручных правок.

Формула сокращения времени:

```text
(manualTime - appTime) / manualTime * 100
```

Формула сокращения действий:

```text
(manualActions - appActions) / manualActions * 100
```

## Порядок проведения

1. Включить экспериментальный режим через переменную окружения.
2. Открыть приложение и выбрать сценарий эксперимента.
3. Ввести обезличенный идентификатор участника при необходимости.
4. Нажать "Начать замер".
5. Выполнить подготовку макетов по сценарию.
6. При необходимости внести ручные правки.
7. Выполнить проверку и экспорт.
8. Нажать "Остановить замер".
9. Добавить результат ручного или шаблонного метода, если проводится сравнение.
10. Экспортировать отчет.

## Критерии готовности макета

Макет считается готовым, если:

- размер соответствует формату;
- текст читаем;
- кнопка с призывом к действию видна;
- изображение не имеет критической обрезки;
- важные элементы находятся в безопасной области;
- нет критических ошибок валидатора;
- файл экспортирован.

## Как включить research mode

Режим включается только через флаг окружения:

```text
VITE_RESEARCH_MODE=true
```

Если флаг не задан или имеет другое значение, экспериментальная панель не отображается, события не записываются, а приложение работает в обычном режиме.

## Как начать и остановить замер

В панели "Режим эксперимента" необходимо выбрать сценарий, метод и нажать "Начать замер". После завершения подготовки нажать "Остановить замер". После остановки отображается блок "Результаты эксперимента" со сводными метриками.

## Как экспортировать результаты

Кнопка "Экспортировать отчет" создает три файла:

- `research-session.json` — полные данные сессии, события и ручные результаты;
- `research-session.csv` — события в формате `timestamp,type,step,formatId,details`;
- `research-summary.txt` — человекочитаемая сводка для анализа.

## Ограничения эксперимента

- Метрики действий отражают действия внутри приложения и не учитывают внешние операции пользователя.
- Время ручного метода вводится исследователем вручную.
- Предупреждения валидатора зависят от текущих правил проверки макета.
- Результаты корректно сравнивать только при одинаковых исходных данных, сценарии и критериях готовности.
- Экспериментальный режим не должен использоваться для сбора персональных данных.
